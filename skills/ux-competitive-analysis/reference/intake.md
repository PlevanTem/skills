# Intent Recognition / 需求补齐（对话式）

> 目标：当用户需求模糊时，先用自然语言把“分析的靶心”定下来。
> 原则：不抛术语、不展示结构化草稿；像设计师做访谈一样问。

## 必须补齐的关键信息（按优先级）
1. **你在服务的业务目标是什么**
   - 例：提升激活/留存/转化；降低学习成本；提高任务完成率；提高付费率；提高信任与可控性
2. **目标用户是谁（1~2 类即可）**
   - 例：新手开发者/资深工程师/企业管理员；是否有代码规范/安全合规要求
3. **核心使用场景/任务流是什么（选 1~3 个）**
   - 例：生成代码、解释代码、重构、调试、写测试、提交 PR、查文档、代码补全
4. **你希望的研究深度**
   - 例：广域盘点为主（列出与分层） vs 深度拆 2~3 个竞品（任务流级别）
5. **你已知或必须覆盖的竞品（如有）**
   - 例：Cursor、GitHub Copilot、Codeium、Tabnine、Amazon Q Developer、JetBrains AI 等
6. **约束条件**
   - 例：必须插件形态/必须离线/必须企业私有化/必须特定 IDE；合规红线；交付时间

## 当用户给的需求很短时，你的开场策略
- 先复述对方目的（用“业务目标 + 场景”语言）
- 然后只问 3~5 个高信息密度问题，把范围锁住
- 得到回答后再进入搜索与拆解

## 你可以直接使用的提问句式（自然语言）
- “这次竞品分析最后要帮助你做什么决策？（比如：选型/定功能/定信息架构/定交互范式）”
- “你的核心用户更偏新手还是熟练开发者？他们最常做的 1~2 个任务是什么？”
- “你最在意的指标是什么？比如首次可用时间、任务完成率、留存、付费转化、满意度？”
- “你希望我偏‘先盘点市场’还是‘深拆 2~3 个关键竞品到任务流’？”
- “有没有必须覆盖/必须避开的竞品或方案？”

## 何时可以开始分析（开始条件）
满足以下任意一种即可开始：
- 用户明确了业务目标 + 1 个核心场景
- 或者用户明确表示“先做广域扫描”，允许你先盘点再回收范围
