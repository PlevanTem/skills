# Analysis Frameworks（拆解框架）

## 一、信息 → 分析 → 结论（闭环模板）
1. **信息（What）**：你看到了什么（界面证据/文档/规则/数据）
2. **分析（Why）**：它为什么这样设计（用户心智、场景约束、业务动机、风险控制、技术边界）
3. **结论（So what / Now what）**：
   - 为什么重要（影响哪个指标/哪个关键时刻）
   - 我们该怎么做（可落地动作 + 验证方式）

> 每条洞察至少满足：有证据、有解释、有建议。

## 二、体验五要素（五层体验）拆解维度
用于把“界面”还原成“体验系统”，避免只评 UI 表面。
1. **战略层**：目标与价值（用户目标/业务目标是否一致）
2. **范围层**：能力边界（功能范围、权限、限制、版本差异）
3. **结构层**：信息架构与交互结构（任务流、导航、信息组织）
4. **框架层**：布局与界面框架（信息优先级、控件策略、密度）
5. **表现层**：视觉与文案（清晰度、信任感、错误提示、可访问性）

## 三、体验设计“六个角度”（落地拆解提示）
- 用户：谁在用？熟练度？
- 场景：何时何地？中断多不多？是否高压？
- 问题：他们要解决什么？最怕什么？
- 交互：步骤数、反馈、容错、默认值、可撤销
- 界面：信息层级、对比度、密度、可扫读性
- 支撑点：策略/权限/合规/模型能力/成本与限制提示

## 四、模式/原则提炼方法（从具体到抽象）
- 先标注“关键界面事实”（截图证据）
- 再归纳“它属于哪类模式”（例如：渐进披露、先给结果后解释、低成本试用、强引导弱打扰）
- 写清“适用条件/不适用边界”
- 转换为“我们可执行的设计原则 + 最小落地动作”
