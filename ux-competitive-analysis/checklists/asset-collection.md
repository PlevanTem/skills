# 视觉证据采集 Checklist（强制）

## 目标
- 每个竞品至少 **3 张有效 UI 截图**（不够就继续搜索/换关键词/换来源）。
- 截图必须服务于论证：能看出关键流程、关键决策点、关键反馈。

## 有效截图标准（满足其一即可）
- 覆盖完整任务流的关键步骤（起点→关键分支→完成/失败）
- 能清晰看到信息架构/布局策略（例如：侧栏结构、主工作区、设置/权限）
- 能清晰看到关键反馈（加载、错误、撤销、确认、成本提示、权限提示等）
- 尽量高清、可读；避免糊图、水印遮挡关键信息

## 推荐截图来源优先级（证据强度从高到低）
1. 官方文档 / 官方博客 / 官方帮助中心（通常更可信）
2. 产品官网的功能页/演示页
3. 权威媒体评测（含发布日期）
4. 视频评测（可截图，但要标注视频来源与时间点）
5. 社区讨论贴（仅用于“摩擦点/抱怨”，不可当作功能事实的唯一证据）

## 命名与落盘（如在支持本地文件系统的环境中）
- 路径：`~/task_name/images/`
- 命名：`{competitor}-{flow}-{step}-{date}.png`
  - 示例：`cursor-chat-refactor-step2-2026-01-07.png`

## 不足 3 张怎么办（强制重试策略）
- 换关键词：加 feature 名、加 “UI”“screenshot”“workflow”“documentation”
- 换渠道：官网/帮助中心/YouTube/评测媒体/论坛
- 换语言：中英双语搜
- 若仍不足：明确在报告中声明“证据不足，以下为推断/待验证”，并降低结论力度
